[[overview]]
== Overview

This extension introduces a unified machine learning (ML) execution
framework for Vulkan.  It defines:

  * A native *tensor resource type* (`VkTensorKHR`) to represent the
    high-dimensionality structured data (up to 8-dimensional) consumed and
    produced by ML workloads.
  * *ML primitive operations* -- IHV-optimized, fixed-function pipeline
    objects encapsulating standard neural-network layers (convolutions,
    matrix multiplications, pooling, and element-wise activations) that
    bypass the overhead of compiling custom compute shaders.
  * An *ML graph* (`VkMLGraphKHR`) abstraction that composes primitive
    operations into a directed acyclic graph (DAG) for whole-model
    dispatch.
  * An *ML session* (`VkMLSessionKHR`) handle that manages transient
    state, scratch memory, and resource bindings for one or more
    concurrent graph evaluations.

The extension integrates natively with existing Vulkan paradigms:

  * Tensor objects follow the create/bind/destroy lifecycle of images
    and buffers.
  * ML graphs are recorded into standard `VkCommandBuffer` objects and
    submitted to compute-capable queues via `vkQueueSubmit2`.
  * Synchronization uses existing pipeline barriers and timeline
    semaphores -- no new synchronization primitives are introduced.
  * Feature and property queries follow the `VkPhysicalDeviceFeatures2`
    / `VkPhysicalDeviceProperties2` pattern.

This extension supersedes vendor-specific proposals including
`VK_ARM_tensors`, `VK_ARM_data_graph`, and the Qualcomm ML primitives
proof-of-concept.  It is designed to serve as the cross-vendor KHR
baseline for ML on Vulkan.

=== Design Goals

  * *Portability*: A single API surface across desktop, mobile, and
    embedded implementations.
  * *Performance*: IHV-provided kernels for primitives, eliminating the
    need for frameworks to ship hand-tuned SPIR-V for each target.
  * *Interoperability*: Tensor resources are accessible from compute
    shaders via SPIR-V, enabling hybrid workloads where custom shaders
    operate alongside opaque ML primitives.
  * *Composability*: ML graphs compose with existing Vulkan compute and
    graphics workloads in the same command buffer, enabling tight
    integration for neural rendering, super-resolution, and inference-
    in-the-loop scenarios.

[[new-object-types]]
== New Object Types

[[VkTensorKHR]]
=== VkTensorKHR

`VkTensorKHR` is a non-dispatchable handle representing an N-dimensional
tensor resource.  Tensors are the fundamental data container for ML
workloads.  A tensor has a format, dimensionality, shape, tiling mode,
and usage flags.  Tensors may be backed by device memory allocated and
bound by the application.

[source,c]
----
VK_DEFINE_NON_DISPATCHABLE_HANDLE(VkTensorKHR)
----


[[VkTensorViewKHR]]
=== VkTensorViewKHR

`VkTensorViewKHR` is a non-dispatchable handle representing a typed view
into a `VkTensorKHR`.  Tensor views enable format reinterpretation and
sub-region access without copying data.

[source,c]
----
VK_DEFINE_NON_DISPATCHABLE_HANDLE(VkTensorViewKHR)
----


[[VkMLGraphKHR]]
=== VkMLGraphKHR

`VkMLGraphKHR` is a non-dispatchable handle representing a compiled
directed acyclic graph (DAG) of ML primitive operations.  Once created,
an ML graph is immutable and may be dispatched multiple times across
command buffers and sessions.

[source,c]
----
VK_DEFINE_NON_DISPATCHABLE_HANDLE(VkMLGraphKHR)
----


[[VkMLSessionKHR]]
=== VkMLSessionKHR

`VkMLSessionKHR` is a non-dispatchable handle representing an execution
context for one or more `VkMLGraphKHR` dispatches.  A session owns
transient scratch memory and intermediate tensor storage required during
graph evaluation.  Multiple sessions may exist concurrently to support
batched or pipelined inference.

[source,c]
----
VK_DEFINE_NON_DISPATCHABLE_HANDLE(VkMLSessionKHR)
----

[[VkPhysicalDeviceMLFeaturesKHR]]
=== VkPhysicalDeviceMLFeaturesKHR

The `VkPhysicalDeviceMLFeaturesKHR` structure is defined as:

[source,c]
----
typedef struct VkPhysicalDeviceMLFeaturesKHR {
    VkStructureType    sType;
    void*              pNext;
    VkBool32           mlPrimitives;
    VkBool32           mlGraph;
    VkBool32           tensorObjects;
    VkBool32           tensorShaderAccess;
    VkBool32           tensorImageAliasing;
    VkBool32           fp16Tensors;
    VkBool32           bf16Tensors;
    VkBool32           int8Tensors;
    VkBool32           int4Tensors;
    VkBool32           fp8Tensors;
    VkBool32           fusedActivations;
    VkBool32           mlGraphScratchAutoAllocation;
} VkPhysicalDeviceMLFeaturesKHR;
----

  * `sType` is a `VkStructureType` value identifying this structure.
  * `pNext` is `NULL` or a pointer to a structure extending this
    structure.
  * `mlPrimitives` indicates whether the implementation supports ML
    primitive operations (convolution, GEMM, pooling, activations).
  * `mlGraph` indicates whether the implementation supports construction
    and dispatch of `VkMLGraphKHR` objects.
  * `tensorObjects` indicates whether the implementation supports
    creation of `VkTensorKHR` objects.
  * `tensorShaderAccess` indicates whether tensors can be accessed from
    compute shaders via SPIR-V `OpTensorReadKHR`/`OpTensorWriteKHR`.
  * `tensorImageAliasing` indicates whether tensors and images may alias
    the same device memory.
  * `fp16Tensors` indicates support for `VK_FORMAT_R16_SFLOAT` tensors.
  * `bf16Tensors` indicates support for `VK_FORMAT_R16_BFLOAT_KHR`
    tensors.
  * `int8Tensors` indicates support for `VK_FORMAT_R8_SINT` /
    `VK_FORMAT_R8_UINT` tensors.
  * `int4Tensors` indicates support for packed 4-bit integer tensors.
  * `fp8Tensors` indicates support for 8-bit floating-point tensors
    (E4M3 and E5M2 encodings).
  * `fusedActivations` indicates whether ML primitives support fused
    activation functions.
  * `mlGraphScratchAutoAllocation` indicates whether the implementation
    can automatically allocate scratch memory for ML graph execution.

This structure *may* be included in the `pNext` chain of
`VkPhysicalDeviceFeatures2` and `VkDeviceCreateInfo`.

.Valid Usage
****
  * [[VUID-VkPhysicalDeviceMLFeaturesKHR-sType-sType]]
    `sType` *must* be
    `VK_STRUCTURE_TYPE_PHYSICAL_DEVICE_ML_FEATURES_KHR`
****


[[VkPhysicalDeviceMLPropertiesKHR]]
=== VkPhysicalDeviceMLPropertiesKHR

The `VkPhysicalDeviceMLPropertiesKHR` structure is defined as:

[source,c]
----
typedef struct VkPhysicalDeviceMLPropertiesKHR {
    VkStructureType    sType;
    void*              pNext;
    uint32_t           maxTensorDimensions;
    uint64_t           maxTensorElements;
    uint32_t           maxTensorDimensionSize;
    uint32_t           maxMLGraphNodeCount;
    uint32_t           maxMLGraphDepth;
    uint32_t           maxMLSessionCount;
    uint32_t           maxConcurrentMLDispatches;
    uint32_t           supportedPrimitiveCount;
    VkDeviceSize       minTensorMemoryAlignment;
    VkDeviceSize       maxScratchMemorySize;
} VkPhysicalDeviceMLPropertiesKHR;
----

  * `sType` is a `VkStructureType` value identifying this structure.
  * `pNext` is `NULL` or a pointer to a structure extending this
    structure.
  * `maxTensorDimensions` is the maximum number of dimensions supported
    for a single tensor.  *Must* be at least 4.
  * `maxTensorElements` is the maximum total number of elements in a
    single tensor.
  * `maxTensorDimensionSize` is the maximum extent of any single tensor
    dimension.
  * `maxMLGraphNodeCount` is the maximum number of nodes in a single
    `VkMLGraphKHR`.
  * `maxMLGraphDepth` is the maximum depth of the DAG in a single
    `VkMLGraphKHR`.
  * `maxMLSessionCount` is the maximum number of `VkMLSessionKHR`
    objects that may exist concurrently on a device.
  * `maxConcurrentMLDispatches` is the maximum number of ML graph
    dispatches that may execute concurrently.
  * `supportedPrimitiveCount` is the number of ML primitive types
    supported by the implementation.
  * `minTensorMemoryAlignment` is the minimum alignment, in bytes, for
    tensor memory allocations.
  * `maxScratchMemorySize` is the maximum amount of scratch memory the
    implementation may require for any single graph dispatch.

This structure *may* be included in the `pNext` chain of
`VkPhysicalDeviceProperties2`.

.Valid Usage
****
  * [[VUID-VkPhysicalDeviceMLPropertiesKHR-sType-sType]]
    `sType` *must* be
    `VK_STRUCTURE_TYPE_PHYSICAL_DEVICE_ML_PROPERTIES_KHR`
****

== Tensor Objects

[[tensor-description]]
=== Tensor Description

The `VkTensorDescriptionKHR` structure describes the geometry, format,
memory layout, and usage of a tensor:

[source,c]
----
typedef struct VkTensorDescriptionKHR {
    VkStructureType          sType;
    const void*              pNext;
    VkTensorTilingKHR        tiling;
    VkFormat                 format;
    uint32_t                 dimensionCount;
    const uint32_t*          pDimensions;
    const VkDeviceSize*      pStrides;
    VkTensorUsageFlagsKHR    usage;
} VkTensorDescriptionKHR;
----

  * `sType` is a `VkStructureType` value identifying this structure.
  * `pNext` is `NULL` or a pointer to a structure extending this
    structure.
  * `tiling` specifies the data layout strategy for tensor elements.
  * `format` is a `VkFormat` describing the type and size of each
    tensor element.  Only single-component formats are permitted.
  * `dimensionCount` is the number of dimensions.  *Must* be between 1
    and `VkPhysicalDeviceMLPropertiesKHR::maxTensorDimensions`
    inclusive.
  * `pDimensions` is a pointer to an array of `dimensionCount` values
    specifying the size of each dimension, ordered from outermost to
    innermost.
  * `pStrides` is `NULL` or a pointer to an array of `dimensionCount`
    values specifying byte strides for each dimension.  If `NULL`, the
    tensor is densely packed.  *Must* be `NULL` when `tiling` is
    `VK_TENSOR_TILING_OPTIMAL_KHR`.
  * `usage` is a bitmask of `VkTensorUsageFlagBitsKHR` describing the
    intended usage of the tensor.

.Valid Usage
****
  * [[VUID-VkTensorDescriptionKHR-dimensionCount-00001]]
    `dimensionCount` *must* be greater than 0 and less than or equal to
    `VkPhysicalDeviceMLPropertiesKHR::maxTensorDimensions`
  * [[VUID-VkTensorDescriptionKHR-pDimensions-00002]]
    Each element of `pDimensions` *must* be greater than 0 and less than
    or equal to
    `VkPhysicalDeviceMLPropertiesKHR::maxTensorDimensionSize`
  * [[VUID-VkTensorDescriptionKHR-pDimensions-00003]]
    The product of all elements of `pDimensions` *must* be less than or
    equal to `VkPhysicalDeviceMLPropertiesKHR::maxTensorElements`
  * [[VUID-VkTensorDescriptionKHR-pStrides-00004]]
    If `tiling` is `VK_TENSOR_TILING_OPTIMAL_KHR`, `pStrides` *must*
    be `NULL`
  * [[VUID-VkTensorDescriptionKHR-format-00005]]
    `format` *must* be a single-component format
  * [[VUID-VkTensorDescriptionKHR-pStrides-00006]]
    If `pStrides` is not `NULL`, each stride *must* be a multiple of
    the element size of `format`
****


[[tensor-tiling]]
=== Tensor Tiling

[source,c]
----
typedef enum VkTensorTilingKHR {
    VK_TENSOR_TILING_OPTIMAL_KHR  = 0,
    VK_TENSOR_TILING_LINEAR_KHR   = 1,
} VkTensorTilingKHR;
----

  * `VK_TENSOR_TILING_OPTIMAL_KHR` -- The implementation selects an
    opaque, device-optimal data layout.  Applications *must not* assume
    any particular arrangement of elements in memory.
  * `VK_TENSOR_TILING_LINEAR_KHR` -- Elements are stored in row-major
    order with application-specified or densely-packed strides.
    Applications *may* map and directly access tensor memory.


[[tensor-usage-flags]]
=== Tensor Usage Flags

[source,c]
----
typedef enum VkTensorUsageFlagBitsKHR {
    VK_TENSOR_USAGE_SHADER_BIT_KHR         = 0x00000001,
    VK_TENSOR_USAGE_TRANSFER_SRC_BIT_KHR   = 0x00000002,
    VK_TENSOR_USAGE_TRANSFER_DST_BIT_KHR   = 0x00000004,
    VK_TENSOR_USAGE_ML_GRAPH_INPUT_BIT_KHR = 0x00000008,
    VK_TENSOR_USAGE_ML_GRAPH_OUTPUT_BIT_KHR= 0x00000010,
    VK_TENSOR_USAGE_ML_GRAPH_WEIGHT_BIT_KHR= 0x00000020,
    VK_TENSOR_USAGE_IMAGE_ALIASING_BIT_KHR = 0x00000040,
} VkTensorUsageFlagBitsKHR;
typedef VkFlags VkTensorUsageFlagsKHR;
----

  * `VK_TENSOR_USAGE_SHADER_BIT_KHR` -- The tensor *may* be accessed
    from compute shaders via SPIR-V tensor accessors.
  * `VK_TENSOR_USAGE_TRANSFER_SRC_BIT_KHR` -- The tensor *may* be used
    as a source in `vkCmdCopyTensorKHR`.
  * `VK_TENSOR_USAGE_TRANSFER_DST_BIT_KHR` -- The tensor *may* be used
    as a destination in `vkCmdCopyTensorKHR`.
  * `VK_TENSOR_USAGE_ML_GRAPH_INPUT_BIT_KHR` -- The tensor *may* be
    bound as an input to an ML graph dispatch.
  * `VK_TENSOR_USAGE_ML_GRAPH_OUTPUT_BIT_KHR` -- The tensor *may* be
    bound as an output of an ML graph dispatch.
  * `VK_TENSOR_USAGE_ML_GRAPH_WEIGHT_BIT_KHR` -- The tensor *may* be
    bound as a constant weight/bias parameter of an ML graph.
  * `VK_TENSOR_USAGE_IMAGE_ALIASING_BIT_KHR` -- The tensor *may* alias
    device memory with a `VkImage` resource.  Requires the
    `tensorImageAliasing` feature.


[[tensor-create-info]]
=== Creating Tensors

To create a tensor, call:

[source,c]
----
VkResult vkCreateTensorKHR(
    VkDevice                        device,
    const VkTensorCreateInfoKHR*    pCreateInfo,
    const VkAllocationCallbacks*    pAllocator,
    VkTensorKHR*                    pTensor);
----

  * `device` is the logical device that creates the tensor.
  * `pCreateInfo` is a pointer to a `VkTensorCreateInfoKHR` structure
    describing the tensor to create.
  * `pAllocator` controls host memory allocation as described in the
    <<memory-allocation,Memory Allocation>> chapter.
  * `pTensor` is a pointer to a `VkTensorKHR` handle in which the
    resulting tensor object is returned.

.Valid Usage
****
  * [[VUID-vkCreateTensorKHR-tensorObjects-00001]]
    The `tensorObjects` feature *must* be enabled
  * [[VUID-vkCreateTensorKHR-device-00002]]
    `device` *must* have been created with at least one queue
****

.Return Codes
--
[cols="1,1"]
|===
| Success | Failure

| `VK_SUCCESS`
| `VK_ERROR_OUT_OF_HOST_MEMORY`

|
| `VK_ERROR_OUT_OF_DEVICE_MEMORY`

|
| `VK_ERROR_VALIDATION_FAILED`
|===
--

The `VkTensorCreateInfoKHR` structure is defined as:

[source,c]
----
typedef struct VkTensorCreateInfoKHR {
    VkStructureType                  sType;
    const void*                      pNext;
    VkTensorCreateFlagsKHR           flags;
    const VkTensorDescriptionKHR*    pDescription;
    VkSharingMode                    sharingMode;
    uint32_t                         queueFamilyIndexCount;
    const uint32_t*                  pQueueFamilyIndices;
} VkTensorCreateInfoKHR;
----

  * `sType` is a `VkStructureType` value identifying this structure.
  * `pNext` is `NULL` or a pointer to a structure extending this
    structure.
  * `flags` is reserved for future use; *must* be 0.
  * `pDescription` is a pointer to a `VkTensorDescriptionKHR` describing
    the tensor geometry, format, and usage.
  * `sharingMode` specifies the sharing mode when the tensor will be
    accessed by multiple queue families.
  * `queueFamilyIndexCount` is the number of entries in
    `pQueueFamilyIndices`.
  * `pQueueFamilyIndices` is a pointer to an array of queue family
    indices that will access this tensor.  Ignored if `sharingMode` is
    not `VK_SHARING_MODE_CONCURRENT`.

.Valid Usage
****
  * [[VUID-VkTensorCreateInfoKHR-sharingMode-00001]]
    If `sharingMode` is `VK_SHARING_MODE_CONCURRENT`,
    `queueFamilyIndexCount` *must* be greater than 1
  * [[VUID-VkTensorCreateInfoKHR-sharingMode-00002]]
    If `sharingMode` is `VK_SHARING_MODE_CONCURRENT`,
    `pQueueFamilyIndices` *must* be a valid pointer to an array of
    `queueFamilyIndexCount` `uint32_t` values, each of which *must*
    be unique and less than the number of queue families available
  * [[VUID-VkTensorCreateInfoKHR-pDescription-00003]]
    `pDescription` *must* be a valid pointer to a valid
    `VkTensorDescriptionKHR` structure
****

[[tensor-memory]]
=== Tensor Memory Requirements and Binding

Applications query tensor memory requirements using
`vkGetTensorMemoryRequirementsKHR` and bind allocated memory using
`vkBindTensorMemoryKHR`.

[source,c]
----
void vkGetTensorMemoryRequirementsKHR(
    VkDevice                                     device,
    const VkTensorMemoryRequirementsInfoKHR*      pInfo,
    VkMemoryRequirements2*                        pMemoryRequirements);
----

  * `device` is the logical device that owns the tensor.
  * `pInfo` is a pointer to a `VkTensorMemoryRequirementsInfoKHR`
    structure identifying the tensor.
  * `pMemoryRequirements` is a pointer to a `VkMemoryRequirements2`
    structure in which the requirements are returned.

[source,c]
----
typedef struct VkTensorMemoryRequirementsInfoKHR {
    VkStructureType    sType;
    const void*        pNext;
    VkTensorKHR        tensor;
} VkTensorMemoryRequirementsInfoKHR;
----

To bind memory to a tensor:

[source,c]
----
VkResult vkBindTensorMemoryKHR(
    VkDevice                              device,
    uint32_t                              bindInfoCount,
    const VkBindTensorMemoryInfoKHR*      pBindInfos);
----

  * `device` is the logical device that owns the tensors.
  * `bindInfoCount` is the number of elements in `pBindInfos`.
  * `pBindInfos` is a pointer to an array of `VkBindTensorMemoryInfoKHR`
    structures.

[source,c]
----
typedef struct VkBindTensorMemoryInfoKHR {
    VkStructureType    sType;
    const void*        pNext;
    VkTensorKHR        tensor;
    VkDeviceMemory     memory;
    VkDeviceSize       memoryOffset;
} VkBindTensorMemoryInfoKHR;
----

.Valid Usage
****
  * [[VUID-VkBindTensorMemoryInfoKHR-tensor-00001]]
    `tensor` *must* not already be bound to memory
  * [[VUID-VkBindTensorMemoryInfoKHR-memoryOffset-00002]]
    `memoryOffset` *must* be a multiple of
    `VkPhysicalDeviceMLPropertiesKHR::minTensorMemoryAlignment`
  * [[VUID-VkBindTensorMemoryInfoKHR-memory-00003]]
    `memory` *must* have been allocated with a memory type compatible
    with the requirements returned by
    `vkGetTensorMemoryRequirementsKHR` for `tensor`
  * [[VUID-VkBindTensorMemoryInfoKHR-size-00004]]
    The size of the memory range starting at `memoryOffset` and ending
    at the end of `memory` *must* be greater than or equal to the
    `size` member of `VkMemoryRequirements` returned by
    `vkGetTensorMemoryRequirementsKHR`
****

.Return Codes
--
[cols="1,1"]
|===
| Success | Failure

| `VK_SUCCESS`
| `VK_ERROR_OUT_OF_HOST_MEMORY`

|
| `VK_ERROR_OUT_OF_DEVICE_MEMORY`
|===

[[tensor-barriers]]
=== Tensor Memory Barriers

[source,c]
----
typedef struct VkTensorMemoryBarrierKHR {
    VkStructureType          sType;
    const void*              pNext;
    VkAccessFlags2           srcAccessMask;
    VkAccessFlags2           dstAccessMask;
    uint32_t                 srcQueueFamilyIndex;
    uint32_t                 dstQueueFamilyIndex;
    VkTensorKHR              tensor;
} VkTensorMemoryBarrierKHR;
----

  * `sType` is a `VkStructureType` value identifying this structure.
  * `pNext` is `NULL` or a pointer to a structure extending this
    structure.
  * `srcAccessMask` is a bitmask of `VkAccessFlagBits2` specifying a
    <<synchronization-access-masks,source access mask>>.
  * `dstAccessMask` is a bitmask of `VkAccessFlagBits2` specifying a
    <<synchronization-access-masks,destination access mask>>.
  * `srcQueueFamilyIndex` is the source queue family for a
    <<synchronization-queue-transfers,queue family ownership transfer>>,
    or `VK_QUEUE_FAMILY_IGNORED`.
  * `dstQueueFamilyIndex` is the destination queue family for a
    queue family ownership transfer, or `VK_QUEUE_FAMILY_IGNORED`.
  * `tensor` is the `VkTensorKHR` affected by this barrier.

The access masks accept the new `VK_ACCESS_2_ML_GRAPH_READ_BIT_KHR`
and `VK_ACCESS_2_ML_GRAPH_WRITE_BIT_KHR` flags in addition to all
existing access flags applicable to compute operations.

To specify one or more tensor memory barriers in a single dependency
operation, the `VkTensorDependencyInfoKHR` structure *may* be included
in the `pNext` chain of `VkDependencyInfo`:

[source,c]
----
typedef struct VkTensorDependencyInfoKHR {
    VkStructureType                     sType;
    const void*                         pNext;
    uint32_t                            tensorMemoryBarrierCount;
    const VkTensorMemoryBarrierKHR*     pTensorMemoryBarriers;
} VkTensorDependencyInfoKHR;
----

  * `sType` is a `VkStructureType` value identifying this structure.
  * `pNext` is `NULL` or a pointer to a structure extending this
    structure.
  * `tensorMemoryBarrierCount` is the number of tensor memory barriers.
  * `pTensorMemoryBarriers` is a pointer to an array of
    `tensorMemoryBarrierCount` `VkTensorMemoryBarrierKHR` structures.

If only a single tensor barrier is required, a `VkTensorMemoryBarrierKHR`
*may* alternatively be chained directly into `VkDependencyInfo::pNext`.

[[convolution-desc]]
=== Convolution Descriptor

[source,c]
----
typedef struct VkMLPrimitiveDescConvolutionKHR {
    VkStructureType              sType;
    const void*                  pNext;
    VkMLTensorLayoutKHR          inputLayout;
    uint32_t                     kernelWidth;
    uint32_t                     kernelHeight;
    uint32_t                     strideX;
    uint32_t                     strideY;
    uint32_t                     dilationX;
    uint32_t                     dilationY;
    VkMLPaddingModeKHR           paddingMode;
    uint32_t                     paddingTop;
    uint32_t                     paddingBottom;
    uint32_t                     paddingLeft;
    uint32_t                     paddingRight;
    uint32_t                     groupCount;
    VkMLActivationFunctionKHR    fusedActivation;
    float                        activationParam0;
    float                        activationParam1;
} VkMLPrimitiveDescConvolutionKHR;
----

  * `inputLayout` specifies the logical ordering of tensor dimensions.
  * `kernelWidth`, `kernelHeight` specify the spatial extent of the
    convolution filter.
  * `strideX`, `strideY` specify the stride of the convolution window.
    *Must* be at least 1.
  * `dilationX`, `dilationY` specify the dilation factor.  *Must* be at
    least 1.
  * `paddingMode` specifies how input is padded.  When
    `VK_ML_PADDING_MODE_EXPLICIT_KHR`, the four padding fields are used.
  * `groupCount` is the number of convolution groups.  A value of 1
    implies standard convolution.  A value equal to the input channel
    count implies depthwise convolution.
  * `fusedActivation` specifies an optional activation applied to the
    output.  Requires `fusedActivations` feature.
  * `activationParam0`, `activationParam1` are parameters for the fused
    activation (e.g., alpha for Leaky ReLU, min/max for Clamp).

.Valid Usage
****
  * [[VUID-VkMLPrimitiveDescConvolutionKHR-strideX-00001]]
    `strideX` and `strideY` *must* be greater than 0
  * [[VUID-VkMLPrimitiveDescConvolutionKHR-dilationX-00002]]
    `dilationX` and `dilationY` *must* be greater than 0
  * [[VUID-VkMLPrimitiveDescConvolutionKHR-paddingMode-00003]]
    If `paddingMode` is not `VK_ML_PADDING_MODE_EXPLICIT_KHR`, the
    padding fields *must* be 0
  * [[VUID-VkMLPrimitiveDescConvolutionKHR-fusedActivation-00004]]
    If `fusedActivation` is not `VK_ML_ACTIVATION_FUNCTION_NONE_KHR`,
    the `fusedActivations` feature *must* be enabled
  * [[VUID-VkMLPrimitiveDescConvolutionKHR-groupCount-00005]]
    `groupCount` *must* evenly divide the input channel dimension
****

[[ml-graph]]
== ML Graph

An ML graph encapsulates a directed acyclic graph of ML primitive
operations compiled for efficient execution on the target device.
Graph compilation is a potentially expensive operation; applications
*should* create graphs at initialization time and reuse them.


[[ml-graph-node]]
=== Graph Node Specification

[source,c]
----
typedef struct VkMLGraphNodeCreateInfoKHR {
    VkStructureType             sType;
    const void*                 pNext;
    VkMLOperationTypeKHR        operationType;
    const void*                 pOperationDesc;
    uint32_t                    inputCount;
    const VkMLTensorBindingKHR* pInputs;
    uint32_t                    outputCount;
    const VkMLTensorBindingKHR* pOutputs;
    const char*                 pNodeName;
} VkMLGraphNodeCreateInfoKHR;
----

  * `operationType` identifies the ML operation performed by this node.
  * `pOperationDesc` is a pointer to an operation-specific descriptor
    structure (e.g., `VkMLPrimitiveDescConvolutionKHR`).  The `sType`
    of the pointed-to structure *must* correspond to `operationType`.
  * `inputCount` is the number of input tensor bindings.
  * `pInputs` is a pointer to an array of `VkMLTensorBindingKHR`
    structures describing input tensors.
  * `outputCount` is the number of output tensor bindings.
  * `pOutputs` is a pointer to an array of `VkMLTensorBindingKHR`
    structures describing output tensors.
  * `pNodeName` is an optional null-terminated UTF-8 string providing a
    debug label.  *May* be `NULL`.

[source,c]
----
typedef enum VkMLTensorBindingTypeKHR {
    VK_ML_TENSOR_BINDING_TYPE_INTERNAL_KHR        = 0,
    VK_ML_TENSOR_BINDING_TYPE_EXTERNAL_INPUT_KHR  = 1,
    VK_ML_TENSOR_BINDING_TYPE_EXTERNAL_OUTPUT_KHR = 2,
    VK_ML_TENSOR_BINDING_TYPE_EXTERNAL_WEIGHT_KHR = 3,
} VkMLTensorBindingTypeKHR;
----

  * `VK_ML_TENSOR_BINDING_TYPE_INTERNAL_KHR` -- The tensor is produced
    by another node in the same graph.  `nodeIndex` identifies the
    producing node.
  * `VK_ML_TENSOR_BINDING_TYPE_EXTERNAL_INPUT_KHR` -- The tensor
    is provided externally as a graph input.  `tensorIndex` indexes
    into `VkMLGraphCreateInfoKHR::pExternalInputDescriptions`.
  * `VK_ML_TENSOR_BINDING_TYPE_EXTERNAL_OUTPUT_KHR` -- The tensor
    is written externally as a graph output.  `tensorIndex` indexes
    into `VkMLGraphCreateInfoKHR::pExternalOutputDescriptions`.
  * `VK_ML_TENSOR_BINDING_TYPE_EXTERNAL_WEIGHT_KHR` -- The tensor
    is provided externally as a constant weight.  `tensorIndex` indexes
    into `VkMLGraphCreateInfoKHR::pConstantWeightDescriptions`.

[source,c]
----
typedef struct VkMLTensorBindingKHR {
    VkStructureType                sType;
    const void*                    pNext;
    VkMLTensorBindingTypeKHR       bindingType;
    uint32_t                       nodeIndex;
    uint32_t                       tensorIndex;
    const VkTensorDescriptionKHR*  pTensorDescription;
} VkMLTensorBindingKHR;
----

  * `sType` is a `VkStructureType` value identifying this structure.
  * `pNext` is `NULL` or a pointer to a structure extending this
    structure.
  * `bindingType` specifies whether this tensor is an internal graph
    edge or an externally-provided input, output, or weight.
  * `nodeIndex` identifies the producing node when `bindingType` is
    `VK_ML_TENSOR_BINDING_TYPE_INTERNAL_KHR`.  Ignored for external
    binding types.
  * `tensorIndex` is the index of the tensor within the producing node's
    output list (for internal bindings), or the index into the
    corresponding external tensor description array (for external
    bindings).
  * `pTensorDescription` describes the tensor geometry and format.
    *Must* be consistent with the description at the referenced
    location.

To dispatch an ML graph for execution, call:

    VkCommandBuffer                     commandBuffer,
    const VkMLGraphDispatchInfoKHR*     pDispatchInfo);
----

  * `commandBuffer` is the command buffer into which the command is
    recorded.
  * `pDispatchInfo` is a pointer to a `VkMLGraphDispatchInfoKHR`
    structure describing the dispatch.

[source,c]
----
typedef struct VkMLGraphDispatchInfoKHR {
    VkStructureType          sType;
    const void*              pNext;
    VkMLSessionKHR           session;
    uint32_t                 inputTensorCount;
    const VkTensorKHR*       pInputTensors;
    uint32_t                 outputTensorCount;
    const VkTensorKHR*       pOutputTensors;
    uint32_t                 weightTensorCount;
    const VkTensorKHR*       pWeightTensors;
} VkMLGraphDispatchInfoKHR;
----

  * `session` is the `VkMLSessionKHR` that manages execution state.
  * `inputTensorCount` *must* equal
    `VkMLGraphCreateInfoKHR::externalInputCount` of the graph
    associated with `session`.
  * `pInputTensors` is a pointer to an array of `VkTensorKHR` handles
    providing input data.
  * `outputTensorCount` *must* equal
    `VkMLGraphCreateInfoKHR::externalOutputCount`.
  * `pOutputTensors` is a pointer to an array of `VkTensorKHR` handles
    receiving output data.
  * `weightTensorCount` *must* equal
    `VkMLGraphCreateInfoKHR::constantWeightCount`.
  * `pWeightTensors` is a pointer to an array of `VkTensorKHR` handles
    providing constant weights.

  * [[VUID-vkCmdDispatchMLGraphKHR-commandBuffer-00001]]
    `commandBuffer` *must* be in the recording state
  * [[VUID-vkCmdDispatchMLGraphKHR-commandBuffer-00002]]

  * [[VUID-vkCmdDispatchMLGraphKHR-session-00003]]
    `session` *must* be a valid `VkMLSessionKHR` handle
  * [[VUID-vkCmdDispatchMLGraphKHR-inputTensorCount-00004]]
    `inputTensorCount` *must* be equal to the `externalInputCount`
    specified when creating the graph associated with `session`
  * [[VUID-vkCmdDispatchMLGraphKHR-outputTensorCount-00005]]
    `outputTensorCount` *must* be equal to the `externalOutputCount`
    specified when creating the graph associated with `session`
  * [[VUID-vkCmdDispatchMLGraphKHR-weightTensorCount-00006]]
    `weightTensorCount` *must* be equal to the `constantWeightCount`
    specified when creating the graph associated with `session`
  * [[VUID-vkCmdDispatchMLGraphKHR-pInputTensors-00007]]
    Each tensor in `pInputTensors` *must* have been created with
    `VK_TENSOR_USAGE_ML_GRAPH_INPUT_BIT_KHR`
  * [[VUID-vkCmdDispatchMLGraphKHR-pOutputTensors-00008]]
    Each tensor in `pOutputTensors` *must* have been created with

  * [[VUID-vkCmdDispatchMLGraphKHR-pWeightTensors-00009]]

    Each tensor in `pInputTensors` *must* have memory bound via
    `vkBindTensorMemoryKHR`
  * [[VUID-vkCmdDispatchMLGraphKHR-pOutputTensors-00011]]
    Each tensor in `pOutputTensors` *must* have memory bound via
    `vkBindTensorMemoryKHR`
  * [[VUID-vkCmdDispatchMLGraphKHR-pWeightTensors-00012]]
    Each tensor in `pWeightTensors` *must* have memory bound via
    `vkBindTensorMemoryKHR`

    corresponding `pExternalInputDescriptions` entry of the graph
  * [[VUID-vkCmdDispatchMLGraphKHR-shapes-00014]]

    corresponding `pExternalOutputDescriptions` entry of the graph
****

This command *must* only be called outside of a render pass instance.


[[execution-model]]
== Execution and Synchronization Model

=== Queue Requirements

ML graph dispatches are submitted to queues that support the
`VK_QUEUE_COMPUTE_BIT` capability.  Implementations *may* additionally
report a dedicated `VK_QUEUE_ML_BIT_KHR` queue family flag (value TBD)
to indicate hardware queues with dedicated ML acceleration resources.

When a queue family reports `VK_QUEUE_ML_BIT_KHR`, dispatching ML
graphs to that queue *may* achieve higher throughput or lower latency
than dispatching to a general-purpose compute queue.

=== Pipeline Stage

The new pipeline stage `VK_PIPELINE_STAGE_2_ML_GRAPH_BIT_KHR`
represents the execution of ML graph dispatches.  This stage is
logically positioned after `VK_PIPELINE_STAGE_2_COMPUTE_SHADER_BIT`
in the pipeline ordering.

=== Synchronization

ML graph dispatches follow existing Vulkan synchronization rules.
Applications use pipeline barriers and/or timeline semaphores to order
ML dispatches with respect to other commands.

==== Intra-command-buffer Dependencies

To ensure that an ML graph dispatch completes before a subsequent
compute shader reads an output tensor:

[source,c]
----
VkTensorMemoryBarrierKHR tensorBarrier = {
    .sType               = VK_STRUCTURE_TYPE_TENSOR_MEMORY_BARRIER_KHR,
    .pNext               = NULL,
    .srcAccessMask       = VK_ACCESS_2_ML_GRAPH_WRITE_BIT_KHR,
    .dstAccessMask       = VK_ACCESS_2_SHADER_READ_BIT,
    .srcQueueFamilyIndex = VK_QUEUE_FAMILY_IGNORED,
    .dstQueueFamilyIndex = VK_QUEUE_FAMILY_IGNORED,
    .tensor              = outputTensor,
};

VkTensorDependencyInfoKHR tensorDepInfo = {
    .sType                    = VK_STRUCTURE_TYPE_TENSOR_DEPENDENCY_INFO_KHR,
    .pNext                    = NULL,
    .tensorMemoryBarrierCount = 1,
    .pTensorMemoryBarriers    = &tensorBarrier,
};

VkDependencyInfo depInfo = {
    .sType                    = VK_STRUCTURE_TYPE_DEPENDENCY_INFO,
    .pNext                    = &tensorDepInfo,
    .dependencyFlags          = 0,
    .memoryBarrierCount       = 0,
    .pMemoryBarriers          = NULL,
    .bufferMemoryBarrierCount = 0,
    .pBufferMemoryBarriers    = NULL,
    .imageMemoryBarrierCount  = 0,
    .pImageMemoryBarriers     = NULL,
};

vkCmdPipelineBarrier2(commandBuffer, &depInfo);
----

The `VkTensorDependencyInfoKHR` is chained into `VkDependencyInfo::pNext`
to supply tensor-specific barriers alongside the existing buffer and
image barrier arrays.

==== Cross-queue Synchronization

When ML graphs execute on a dedicated ML queue and results are consumed
on a graphics or compute queue, a timeline semaphore *must* be used:

[source,c]
----
VkSemaphoreSubmitInfo mlSignal = {
    .sType      = VK_STRUCTURE_TYPE_SEMAPHORE_SUBMIT_INFO,
    .semaphore  = timelineSemaphore,
    .value      = mlCompleteValue,
    .stageMask  = VK_PIPELINE_STAGE_2_ML_GRAPH_BIT_KHR,
};
// Signal after ML dispatch on ML queue

VkSemaphoreSubmitInfo computeWait = {
    .sType      = VK_STRUCTURE_TYPE_SEMAPHORE_SUBMIT_INFO,
    .semaphore  = timelineSemaphore,
    .value      = mlCompleteValue,
    .stageMask  = VK_PIPELINE_STAGE_2_COMPUTE_SHADER_BIT,
};
// Wait before compute dispatch on compute queue
----

==== Tensor Queue Family Ownership Transfer

When a tensor is accessed by multiple queue families with
`VK_SHARING_MODE_EXCLUSIVE`, a queue family ownership transfer *must*
be performed using paired `VkTensorMemoryBarrierKHR` operations with
appropriate `srcQueueFamilyIndex` and `dstQueueFamilyIndex`.

=== Implicit Ordering Guarantees

Within a single `vkCmdDispatchMLGraphKHR` invocation, the implementation
*must* respect all data dependencies defined by the edges of the ML
graph.  No explicit synchronization is required between nodes of the
same graph dispatch.

Multiple `vkCmdDispatchMLGraphKHR` calls within the same command buffer
are *not* implicitly ordered; a pipeline barrier *must* be inserted if
the output of one dispatch is consumed as input to a subsequent dispatch.

[[interactions]]
== Interactions with Other Extensions

=== VK_KHR_cooperative_matrix

When both `VK_KHR_ml_primitives` and `VK_KHR_cooperative_matrix` are
supported, the GEMM and fully-connected ML primitives *should* be
implemented using cooperative matrix hardware where available.  The
implementation *may* fall back to alternative strategies when the
requested matrix dimensions or data types are not supported by the
cooperative matrix path.

Applications that require fine-grained control over cooperative matrix
tile sizes *should* use `VK_KHR_cooperative_matrix` directly via
compute shaders, loading from `VkTensorKHR` resources through SPIR-V
tensor accessors.

=== VK_KHR_deferred_host_operations

`vkCreateMLGraphKHR` *may* be deferred using
`VK_KHR_deferred_host_operations` when that extension is enabled.
Graph compilation can be an expensive operation, and deferral allows
applications to perform compilation on background threads.

=== IREE and Compiler-based ML Stacks

Compiler-based ML stacks such as IREE *may* target `VK_KHR_ml_primitives`
in one of two modes:

  * *Primitive mode*: The compiler maps each supported ML operation to
    the corresponding `VkMLOperationTypeKHR` primitive, constructing a
    `VkMLGraphKHR` at runtime.  This mode achieves maximum IHV
    optimization.
  * *Hybrid mode*: Unsupported operations are lowered to SPIR-V compute
    shaders that read and write `VkTensorKHR` resources, while supported
    operations use primitives.  Pipeline barriers synchronize between
    the two execution domains.

The SPIR-V `OpTypeTensorKHR` type system enables both modes to share
tensor resources without intermediate copies.


[[usage-example]]
== Usage Example

The following C++ example demonstrates creating two tensors, building a
single-layer convolution graph, and dispatching it.

[source,cpp]
----
#include <vulkan/vulkan.h>
#include <array>
#include <vector>
#include <cassert>
#include <cstring>

// Assumes device, physicalDevice, computeQueue, commandPool, and
// allocator are already initialized.
// Error checking abbreviated for brevity; production code must check
// every VkResult.

// ---------------------------------------------------------------
// 1. Query ML support
// ---------------------------------------------------------------
VkPhysicalDeviceMLFeaturesKHR mlFeatures = {
    .sType = VK_STRUCTURE_TYPE_PHYSICAL_DEVICE_ML_FEATURES_KHR,
};
VkPhysicalDeviceFeatures2 features2 = {
    .sType = VK_STRUCTURE_TYPE_PHYSICAL_DEVICE_FEATURES_2,
    .pNext = &mlFeatures,
};
vkGetPhysicalDeviceFeatures2(physicalDevice, &features2);
assert(mlFeatures.tensorObjects && mlFeatures.mlPrimitives &&
       mlFeatures.mlGraph && mlFeatures.fusedActivations);

// ---------------------------------------------------------------
// 2. Create input tensor: NCHW [1, 3, 224, 224], FP16
// ---------------------------------------------------------------
const std::array<uint32_t, 4> inputDims = {1, 3, 224, 224};
VkTensorDescriptionKHR inputDesc = {
    .sType          = VK_STRUCTURE_TYPE_TENSOR_DESCRIPTION_KHR,
    .pNext          = nullptr,
    .tiling         = VK_TENSOR_TILING_OPTIMAL_KHR,
    .format         = VK_FORMAT_R16_SFLOAT,
    .dimensionCount = 4,
    .pDimensions    = inputDims.data(),
    .pStrides       = nullptr,
    .usage          = VK_TENSOR_USAGE_ML_GRAPH_INPUT_BIT_KHR |
                      VK_TENSOR_USAGE_TRANSFER_DST_BIT_KHR,
};

VkTensorCreateInfoKHR inputCI = {
    .sType       = VK_STRUCTURE_TYPE_TENSOR_CREATE_INFO_KHR,
    .pNext       = nullptr,
    .flags       = 0,
    .pDescription = &inputDesc,
    .sharingMode = VK_SHARING_MODE_EXCLUSIVE,
};

VkTensorKHR inputTensor;
vkCreateTensorKHR(device, &inputCI, nullptr, &inputTensor);

// Query and bind memory
VkTensorMemoryRequirementsInfoKHR memReqInfo = {
    .sType  = VK_STRUCTURE_TYPE_TENSOR_MEMORY_REQUIREMENTS_INFO_KHR,
    .pNext  = nullptr,
    .tensor = inputTensor,
};
VkMemoryRequirements2 memReqs = {
    .sType = VK_STRUCTURE_TYPE_MEMORY_REQUIREMENTS_2,
    .pNext = nullptr,
};
vkGetTensorMemoryRequirementsKHR(device, &memReqInfo, &memReqs);

VkMemoryAllocateInfo allocInfo = {
    .sType           = VK_STRUCTURE_TYPE_MEMORY_ALLOCATE_INFO,
    .pNext           = nullptr,
    .allocationSize  = memReqs.memoryRequirements.size,
    .memoryTypeIndex = /* select device-local type from memReqs */ 0,
};
VkDeviceMemory inputMemory;
vkAllocateMemory(device, &allocInfo, nullptr, &inputMemory);

VkBindTensorMemoryInfoKHR bindInfo = {
    .sType        = VK_STRUCTURE_TYPE_BIND_TENSOR_MEMORY_INFO_KHR,
    .pNext        = nullptr,
    .tensor       = inputTensor,
    .memory       = inputMemory,
    .memoryOffset = 0,
};
vkBindTensorMemoryKHR(device, 1, &bindInfo);

// ---------------------------------------------------------------
// 3. Create weight tensor: [64, 3, 3, 3] (64 filters, 3x3 kernel)
// ---------------------------------------------------------------
const std::array<uint32_t, 4> weightDims = {64, 3, 3, 3};
VkTensorDescriptionKHR weightDesc = {
    .sType          = VK_STRUCTURE_TYPE_TENSOR_DESCRIPTION_KHR,
    .pNext          = nullptr,
    .tiling         = VK_TENSOR_TILING_OPTIMAL_KHR,
    .format         = VK_FORMAT_R16_SFLOAT,
    .dimensionCount = 4,
    .pDimensions    = weightDims.data(),

    .usage          = VK_TENSOR_USAGE_ML_GRAPH_WEIGHT_BIT_KHR |
                      VK_TENSOR_USAGE_TRANSFER_DST_BIT_KHR,

    .paddingBottom   = 0,
    .paddingLeft     = 0,

// ---------------------------------------------------------------
VkCommandBufferAllocateInfo cmdAllocInfo = {
    .sType              = VK_STRUCTURE_TYPE_COMMAND_BUFFER_ALLOCATE_INFO,

    .level              = VK_COMMAND_BUFFER_LEVEL_PRIMARY,
    .commandBufferCount = 1,
};
VkCommandBuffer cmdBuf;
vkAllocateCommandBuffers(device, &cmdAllocInfo, &cmdBuf);

VkCommandBufferBeginInfo beginInfo = {
    .sType = VK_STRUCTURE_TYPE_COMMAND_BUFFER_BEGIN_INFO,

};

VkMLGraphDispatchInfoKHR dispatchInfo = {
    .sType            = VK_STRUCTURE_TYPE_ML_GRAPH_DISPATCH_INFO_KHR,
    .pNext            = nullptr,
    .session          = session,
    .inputTensorCount = 1,
    .pInputTensors    = &inputTensor,
    .outputTensorCount = 1,
    .pOutputTensors   = &outputTensor,
    .weightTensorCount = 1,
    .pWeightTensors   = &weightTensor,
};

vkCmdDispatchMLGraphKHR(cmdBuf, &dispatchInfo);

// Insert barrier: ML write -> transfer read
VkTensorMemoryBarrierKHR barrier = {
    .sType               = VK_STRUCTURE_TYPE_TENSOR_MEMORY_BARRIER_KHR,
    .pNext               = nullptr,
    .srcAccessMask       = VK_ACCESS_2_ML_GRAPH_WRITE_BIT_KHR,
    .dstAccessMask       = VK_ACCESS_2_TRANSFER_READ_BIT,
    .srcQueueFamilyIndex = VK_QUEUE_FAMILY_IGNORED,
    .dstQueueFamilyIndex = VK_QUEUE_FAMILY_IGNORED,
    .tensor              = outputTensor,
};
VkTensorDependencyInfoKHR tensorDepInfo = {
    .sType                    = VK_STRUCTURE_TYPE_TENSOR_DEPENDENCY_INFO_KHR,
    .pNext                    = nullptr,
    .tensorMemoryBarrierCount = 1,
    .pTensorMemoryBarriers    = &barrier,
};
VkDependencyInfo depInfo = {
    .sType                    = VK_STRUCTURE_TYPE_DEPENDENCY_INFO,
    .pNext                    = &tensorDepInfo,
    .dependencyFlags          = 0,
    .memoryBarrierCount       = 0,
    .pMemoryBarriers          = nullptr,
    .bufferMemoryBarrierCount = 0,
    .pBufferMemoryBarriers    = nullptr,
    .imageMemoryBarrierCount  = 0,
    .pImageMemoryBarriers     = nullptr,
};
vkCmdPipelineBarrier2(cmdBuf, &depInfo);

vkEndCommandBuffer(cmdBuf);

// Submit with timeline semaphore signaling
VkSemaphoreSubmitInfo signalInfo = {
    .sType     = VK_STRUCTURE_TYPE_SEMAPHORE_SUBMIT_INFO,
    .semaphore = timelineSemaphore,
    .value     = 1,
    .stageMask = VK_PIPELINE_STAGE_2_ML_GRAPH_BIT_KHR,
};
VkCommandBufferSubmitInfo cmdBufInfo = {
    .sType         = VK_STRUCTURE_TYPE_COMMAND_BUFFER_SUBMIT_INFO,
    .commandBuffer = cmdBuf,
};
VkSubmitInfo2 submitInfo = {
    .sType                    = VK_STRUCTURE_TYPE_SUBMIT_INFO_2,
    .commandBufferInfoCount   = 1,
    .pCommandBufferInfos      = &cmdBufInfo,
    .signalSemaphoreInfoCount = 1,
    .pSignalSemaphoreInfos    = &signalInfo,
};
vkQueueSubmit2(computeQueue, 1, &submitInfo, VK_NULL_HANDLE);
